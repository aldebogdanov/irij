* Preamble: Design Axioms

  Irij (ℑ, ASCII =IR=) is a language designed
  around five non-negotiable axioms that inform every syntax and semantics
  decision:

  1. *AI-generates, human-audits.* The primary author is an LLM. Syntax
     optimizes for token count and generation reliability, not typing comfort.
  2. *Effects are the universal joint.* Concurrency, IO, choreography,
     capabilities — all are algebraic effects. One mechanism, composable
     everywhere.
  3. *Verification is a dial, not a switch.* From dynamic through contracts,
     property tests, refinement types, to proofs — per-binding, not per-project.
  4. *Distribution is syntax, not infrastructure.* Participants, communication,
     and projection are first-class constructs, not library patterns.
  5. *Code is data is hash.* Every definition is content-addressed. No builds,
     no stale state, perfect incrementalism.

* Part I: Syntax Philosophy — "Dense ASCII"

** 1.1 Design Rationale

   Research shows a *2.6× token gap* between the most and least efficient
   languages under BPE tokenization. The winners share traits: ASCII-only
   symbols, minimal punctuation, implicit structure, and few reserved words.

   Irij invents a notation we call *Dense ASCII*: a post-fix/infix hybrid using
   two-character ASCII digraphs for all built-in operations — no Unicode, no
   deep nesting, no ceremony.

   Core principles:
   - *Flat over nested:* max 2 levels of syntactic nesting in normal code.
     Deep nesting is a code smell the formatter warns about. (KV-cache
     friendly for transformer attention.)
   - *Digraph operators:* all built-in operations are 1–2 ASCII characters.
     User-defined operators are spelled out as words.
   - *Significant whitespace for grouping:* indentation defines blocks
     (like Python/Haskell), eliminating brace/bracket tokens.
   - *Pipeline is king:* left-to-right data flow via pipe operator =|>=,
     mirroring how both humans and LLMs read sequentially.
   - *Keywords are short words, not digraphs.* A small set of reserved
     words (=fn=, =do=, =if=, =else=, =match=, =type=, =newtype=,
     =mod=, =use=, =pub=, =with=, =scope=, =effect=, =role=, =cap=,
     =handler=, =impl=, =proto=, =pre=, =post=, =law=, =contract=,
     =select=, =enclave=, =forall=, =par-each=, =on-failure=) are
     2–10 characters each —
     typically one BPE token. These are distinct from operators:
     operators act on values, keywords introduce structural forms.
     Keeping keywords as readable words costs negligible tokens
     while dramatically improving auditability.

** 1.2 Canonical Form: Indentation, Not S-Expressions

   *Indentation is the canonical form* — the primary representation that
   LLMs generate and humans read.

   Rationale: LLMs generate tokens left-to-right. Maintaining a balanced
   parenthesis stack is a known failure mode for transformers, especially
   at depth > 3. Indentation requires only "repeat the whitespace prefix"
   which is trivially reliable. Indentation is also more token-efficient:
   whitespace tokens are cheap (1 token per indent level) vs. open-paren
   + close-paren (2 extra tokens per nesting level).

   S-expression form is supported as a *debug/interchange* format — useful
   for AST serialization, tooling, and as a fallback when the indentation
   parser encounters ambiguity. The compiler can convert between forms,
   but indentation is the default and the generation target.

   #+begin_example
   ;; ── Canonical form (indentation) ─────────────────────────
   fn factorial :: Int -> Int
     0 => 1
     n => n * factorial (n - 1)

   ;; ── Debug/interchange form (S-expression) ────────────────
   ;; Used for: AST dumps, tooling interchange, edge cases.
   ;; NOT the generation target for LLMs.
   (fn factorial (:: Int (-> Int))
     (=> 0 1) (=> n (* n (factorial (- n 1)))))
   #+end_example

   The compiler accepts both forms and can convert between them.
   Note that S-expressions are written *flat* (no internal indentation)
   — if you need indentation to read them, use the canonical form instead.

** 1.3 Lexical Grammar Sketch

*** 1.3.1 Identifiers & Literals

    #+begin_example
    ;; identifiers: lowercase kebab-case (most token-efficient)
    my-value
    http-get
    user->json

    ;; type names: PascalCase
    HttpResponse
    UserRecord

    ;; role/location names: $-prefixed UPPER-CASE (choreography)
    $ALICE
    $BOB
    $DB-PRIMARY

    ;; numeric: standard with underscore separators
    42  3.14  0xFF  1_000_000  2/3

    ;; strings: double-quote, interpolation with ${}
    "hello ${name}"

    ;; keywords (like Clojure): colon-prefix atoms
    :ok  :error  :pending

    ;; comments: double-semicolon to end of line
    ;; this is a comment
    #+end_example

*** 1.3.2 Core Digraph Operators

    The operator table below is the heart of token efficiency. Each digraph
    is chosen to be (a) mnemonic, (b) a single BPE token under GPT-4/Claude
    tokenizers, and (c) visually distinct.

    #+begin_example
    ;; ── Binding & Definition ────────────────────────────────
    :=    ;; immutable binding        x := 42
    :!    ;; mutable binding          x :! 0
    <-    ;; mutation / assignment     x <- x + 1
    ->    ;; function arrow            (x y -> x + y)
    =>    ;; fat arrow / block intro    :ok v => v  OR  => body
    ::    ;; type annotation           x :: Int
    =     ;; map field separator       {name= "jo"}

    ;; ── Pipeline & Composition ──────────────────────────────
    |>    ;; pipe forward              xs |> map f |> filter g
    <|    ;; pipe backward             print <| compute x
    >>    ;; compose forward           f >> g  ≡  (x -> x |> f |> g)
    <<    ;; compose backward          g << f

    ;; ── Arithmetic & Logic ──────────────────────────────────
    + - * /          ;; standard
    %                ;; modulo
    **               ;; power
    == /=            ;; equal, not-equal
    < > <= >=        ;; comparison
    && || !          ;; boolean
    ++               ;; concatenation (strings, lists, etc.)

    ;; ── Collection / Array ──────────────────────────────────
    #[  ]            ;; vector literal         #[1 2 3]
    #{ }             ;; set literal            #{:a :b}
    #( )             ;; tuple literal          #(1 "a" :ok)
    { }              ;; map literal            {name= "jo" age= 5}
    ..               ;; range                  1..10
    ..<              ;; exclusive range        0 ..< n

    ;; ── Array / Sequence Operations (J-inspired digraphs) ──
    /+    ;; reduce with +             #[1 2 3] |> /+       ;; => 6
    /*    ;; reduce with *             #[1 2 3] |> /*       ;; => 6
    /#    ;; count                     #[1 2 3] |> /#       ;; => 3
    /&    ;; all? (reduce with &&)
    /|    ;; any? (reduce with ||)
    /^    ;; generic reduce (fold)     #[1 2 3] |> /^ (+)   ;; => 6
    /$    ;; scan (prefix sums)        #[1 2 3] |> /$ (+)   ;; => #[1 3 6]
    @     ;; map-over / each           xs |> @ f
    @i    ;; map-indexed               xs |> @i (i v -> ...)
    /?    ;; filter                    xs |> /? even?
    /!    ;; first match / find        xs |> /! (> 3)

    ;; ── Operator Sections (operators as first-class values) ──
    (+)   ;; addition as function      (#[1 2 3] |> /^ (+))
    (-)   ;; subtraction as function
    (*)   ;; multiplication as function
    (==)  ;; equality as function

    ;; ── Effect & Control ────────────────────────────────────
    :::   ;; effect separator           fn f :: A B ::: E1 E2
    do    ;; sequence effects           do (print "a") (print "b")
    if    ;; conditional                if cond then-expr else-expr
    match ;; pattern match              match x (:ok v => ...) (:err e => ...)
    with  ;; effect handler             with handler expr

    ;; ── Choreography ────────────────────────────────────────
    ~>    ;; send                      val ~> $ROLE
    <~    ;; receive (sugar)           val <~ $ROLE
    ~*>   ;; broadcast                 val ~*> #[$R1 $R2]
    ~/    ;; select (inform branch)    decision ~/ $ROLE
    @     ;; located-at (types & exprs) Str @$BUYER  match cmd @$PRIMARY
    #+end_example

*** 1.3.3 Token Budget Comparison

    The following implements "map a function over a list, filter, reduce":

    #+begin_example
    ;; Irij: ~12 tokens
    xs |> @ f |> /? g |> /+

    ;; Clojure: ~18 tokens
    (->> xs (map f) (filter g) (reduce +))

    ;; Python: ~22 tokens
    reduce(lambda a,b: a+b, filter(g, map(f, xs)))

    ;; Java: ~35 tokens
    xs.stream().map(f).filter(g).reduce(0, Integer::sum);
    #+end_example

** 1.4 Block Structure & Indentation

   Blocks are defined by indentation (strict 2-space). The keyword that
   opens a block is on the same line; the body is indented below.

   *Strict 2-space rule:* Indentation MUST be exactly 2 spaces per level.
   The lexer treats =  = (two ASCII spaces) as a single INDENT token.
   4 spaces = indent level 2. Odd or non-2-space indentation is a *hard
   parse error*. This is not a style preference — it is a grammar rule,
   analogous to a digraph. Benefits: zero style variance across codebases,
   deterministic generation by LLMs, minimal token cost.

   #+begin_example
   ;; function definition (canonical form)
   fn factorial :: Int -> Int
     0 => 1
     n => n * factorial (n - 1)
   #+end_example

** 1.5 Expression Termination & Inline Sequencing

   In function bodies, *newlines separate statements*. Each indented line
   at the same level is a separate statement.

   *Semicolons* (=;=) are the expression separator for *inline sequencing*,
   but they are only permitted inside parenthesized expressions:

   #+begin_example
   ;; semicolons OK inside parentheses
   s.fork (-> cmd ~> $REPLICA; $REPLICA.store k v)

   ;; NOT allowed at top level — use newlines instead
   ;; WRONG: name := read-line (); print name
   ;; RIGHT:
   name := read-line ()
   print name
   #+end_example

   *Line continuation* uses =\= (backslash) at end of line. A trailing =\=
   joins the following line into the current expression:

   #+begin_example
   ;; explicit continuation with \
   long-result := compute-something a b \
     + adjust c \
     + finalize d

   ;; implicit continuation: a more-indented line starting with a
   ;; binary operator automatically continues the previous expression
   result := data
     |> parse
     |> validate
     |> transform
   #+end_example

   *Implicit continuation rule:* if a line is more indented than the
   current statement and begins with a binary operator (=|>=, =+= ,
   =&&=, =|=, etc.), it is a continuation of the previous line. The =\=
   explicit marker is only needed when the continuation line does NOT
   start with an operator.

* Part II: Core Language Constructs

** 2.1 Bindings, Functions & Closures

   #+begin_example
   ;; immutable binding
   x := 42

   ;; with type annotation
   x := 42 :: Int

   ;; mutable binding (requires Mut effect)
   x :! 0 :: Int

   ;; anonymous function (lambda)
   (x y -> x + y)

   ;; ── Three function body forms ────────────────────────

   ;; 1. Lambda — single expression, params bound with ->
   fn add :: Int -> Int -> Int
     (x y -> x + y)

   ;; 2. Pattern-match arms — each arm matches the argument
   fn fib :: Int -> Int
     0 => 0
     1 => 1
     n => fib (n - 1) + fib (n - 2)

   ;; 3. Imperative block — => binds params, body is multi-statement
   ;;    => alone means the function takes ()
   ;;    => params binds positional arguments from the type signature
   fn greet :: Str () ::: Console
     => name
     print "Hello, ${name}!"

   fn main :: () () ::: IO
     =>
     config := read-config ()
     run-server config

   ;; partial application is automatic (currying)
   add-one := add 1      ;; Int -> Int

   ;; point-free composition
   process := parse >> validate >> transform >> serialize
   #+end_example

   *Function body disambiguation:* the parser determines the body form
   by the first token after the indent:
   - =(= → lambda (expression in parentheses with =->= arrow)
   - =pattern => expr= → pattern-match arm (one or more arms)
   - ==>= alone or ==> params= → imperative block (multi-statement body)

   *Mutable bindings:* =:!= creates, =<-= writes, reading is by name:

   #+begin_example
   x :! 0          ;; create mutable binding
   x <- x + 1      ;; mutate
   print x          ;; read: just use the name
   #+end_example

** 2.2 Data Types & Records

   #+begin_example
   ;; product type (record) — fields declared with ::
   type User
     name :: Str
     age  :: Int
     role :: Role

   ;; construction: positional args in field-declaration order
   u := User "Jo" 30 :admin     ;; → User {name= Jo age= 30 role= :admin}

   ;; field access via dot
   u.name   ;; → "Jo"
   u.age    ;; → 30

   ;; sum type (tagged union)
   type Result a e
     Ok  a
     Err e

   ;; parameterized type
   type Tree a
     Leaf a
     Node (Tree a) a (Tree a)

   ;; newtype (zero-cost wrapper)
   newtype Email := Str

   ;; anonymous record (structural typing)
   person := {name= "Jo" age= 30}
   #+end_example

** 2.3 Pattern Matching & Destructuring

   #+begin_example
   ;; match expression
   match result
     Ok value  => process value
     Err :timeout => retry
     Err other => do (log other) fail

   ;; destructuring in bindings — works on both anonymous records and product types
   {name= n age= a} := get-user id          ;; anonymous record
   {name= n age= a} := User "Jo" 30 :admin  ;; product type (same syntax)

   ;; product type patterns — named (partial) or positional (all fields)
   match user
     {name= "Jo"}  => println "found Jo"  ;; named: match any subset of fields
     {name= n}     => println n            ;; named: bind field to variable
     User n a _    => println n            ;; positional: all fields, in order

   ;; guards
   match score
     s | s >= 90 => :excellent
     s | s >= 70 => :good
     _           => :needs-work

   ;; list patterns (... for rest, distinct from .. range operator)
   match xs
     #[]       => :empty
     #[x]      => :singleton x
     #[x ...r] => :cons x r      ;; head + rest (three dots = spread)

   ;; match as expression — usable on the right side of :=
   label := match score
     s | s >= 90 => "excellent"
     s | s >= 70 => "good"
     _           => "needs-work"
   #+end_example

   Note: =...= (three dots) is the spread/rest operator in patterns,
   distinct from =..= (two dots) which is the range operator in
   expressions. This avoids the ambiguity of using the same symbol
   for both purposes.

* Part III: Algebraic Effects — The Universal Joint

** 3.1 Effect Declaration

   Effects are the *only* mechanism for side effects, concurrency, IO,
   choreography, and capabilities. A function's type signature declares
   exactly which effects it may perform.

   *Lineage:* Algebraic effects originate from Plotkin & Power (2003) and
   were made practical in Koka (Microsoft Research), Eff (Pretnar),
   OCaml 5, and Unison. The key idea: a function declares *what*
   operations it needs (e.g., "I need to print") without defining *how*
   they work. The "how" is provided separately by a handler (§3.2).
   This is like dependency injection, but tracked in the type system.

   #+begin_example
   ;; declaring an effect = declaring an interface of operations
   ;; (think: "this is what I might do")
   effect Console
     print :: Str -> ()
     read-line :: () -> Str

   effect FileIO
     read-file  :: Path -> Bytes
     write-file :: Path -> Bytes -> ()

   effect Http
     fetch :: Request -> Response

   ;; function with effects declared
   fn greet :: Str () ::: Console
     => name
     print "Hello, ${name}!"

   ;; multiple effects
   fn sync-data :: () (Result Data Error) ::: Http FileIO Log

   ;; pure function (no effects — the default, no annotation needed)
   fn add :: Int -> Int -> Int
     (x y -> x + y)
   #+end_example

   The effect declaration =::: E1 E2= after the type signature lists which
   effects the function may perform. No annotation means *pure*. =::= is
   for data types (input/output schemas), =:::= is for effects.

** 3.2 Effect Handlers

   Handlers intercept effect operations and define their concrete behavior.
   They are first-class values, composable, and lexically scoped.

   *How it works:* When a function performs an effect operation (e.g.,
   =print "hello"=), execution pauses at that point and control transfers
   to the enclosing handler. The handler executes its implementation for
   that operation (e.g., writes to stdout), then *resumes* the original
   function from where it paused. This is like =try/catch= but the catch
   can return a value back to the throw site and continue execution.

   *Why this matters:* The same function can be run with completely
   different effect implementations without changing its code. Production
   handler writes to real stdout; test handler collects strings into a
   list. The function under test doesn't know the difference.

   #+begin_example
   ;; handler definition: provides concrete behavior for an effect
   handler console-to-stdout :: Console
     print msg => io.stdout.write msg       ;; "print" means write to stdout
     read-line () => io.stdin.read-line ()  ;; "read-line" means read stdin

   ;; using a handler: wraps a block, intercepts its effects
   with console-to-stdout
     greet "world"      ;; greet's print/read-line go to real stdout/stdin

   ;; test handler: same effect, completely different behavior (pure!)
   handler mock-console :: Console
     state :! #["Alice"]    ;; mutable binding within handler
     print msg => state <- state ++ #[msg]   ;; "print" appends to list
     read-line () =>                          ;; "read-line" pops from list
       result := head state                  ;; read mutable by name
       state <- tail state                   ;; write with <-
       result

   ;; now test greet without any real IO:
   with mock-console
     greet "world"      ;; greet's print/read-line go to in-memory list

   ;; composing handlers: stack multiple effect implementations
   with (mock-console >> file-to-memory >> http-to-cache)
     sync-data ()        ;; all three effects handled, fully pure
   #+end_example

** 3.3 Capabilities as Effects

   A capability is an unforgeable permission token that grants the right to
   perform certain effects. This combines effect tracking with a security
   model.

   *Lineage:* From Austral's linear capabilities and the Effekt language's
   insight that "tracking capabilities is equivalent to tracking effects."

   *How it works:* A capability is declared like an effect but with =cap=.
   Functions that need it must declare it in their effect row. The
   capability is created at a specific scope (e.g., when connecting to a
   database) and ceases to exist when that scope exits. The runtime tracks
   capabilities *linearly* — they cannot be duplicated, stored globally,
   or leaked to unauthorized code.

   *Why this matters:* A function's type signature reveals exactly what
   resources it can access. If =migrate= requires =[DbWrite Log]=, you
   know it can write to the database and log, and *nothing else* — it
   cannot make HTTP calls, read files, or access other databases. This
   provides supply-chain-attack resistance: a dependency that claims to
   only need =Log= literally cannot exfiltrate data over the network.

   #+begin_example
   ;; capability declaration: like an effect, but permission-gated
   cap DbWrite :: effect
     insert :: Table -> Row -> ()
     update :: Table -> Query -> Row -> ()
     delete :: Table -> Query -> ()

   ;; function requiring capability: must receive it from caller
   fn migrate :: () () ::: DbWrite Log
     ...

   ;; granting capability: scoped, cannot escape the with-block
   fn main :: () () ::: IO
     =>
     with (db-connect "postgres://..." |> grant DbWrite)
       migrate ()
     ;; DbWrite capability does not exist here — cannot be leaked
   #+end_example

* Part IV: Structured Concurrency

** 4.1 Design: Concurrency as Effects

   All concurrency primitives are algebraic effects. The scheduler itself is
   a handler. This means concurrency semantics are *user-definable* — you
   can swap the scheduler by swapping the handler.

   #+begin_example
   ;; built-in concurrency effect
   effect Async
     fork   :: (() -> a) -> Fiber a
     await  :: Fiber a -> a
     sleep  :: Duration -> ()
     cancel :: Fiber a -> ()
   #+end_example

** 4.2 Nursery / Scope Primitive

   The =scope= construct is the structured concurrency primitive. All fibers
   forked within a scope must complete (or be cancelled) before the scope
   exits. This is enforced by the compiler — there is no =spawn-and-forget=.

   For the rare case where "fire-and-forget" is genuinely needed (e.g.,
   async replication after a client has already received a response), the
   =detach!= escape hatch explicitly opts out of structured concurrency.
   The =!= suffix signals danger — this is intentionally loud syntax for
   an intentionally dangerous operation.

   #+begin_example
   ;; basic structured concurrency
   fn fetch-dashboard :: Id Dashboard ::: Http Async
     => id
     scope s
       user-f   := s.fork (-> fetch-user id)
       orders-f := s.fork (-> fetch-orders id)
       stats-f  := s.fork (-> fetch-stats id)
       ;; all three run concurrently
       ;; scope waits for all before continuing
       make-dashboard (await user-f) (await orders-f) (await stats-f)

   ;; racing — first result wins, others cancelled
   fn fetch-fastest :: () Data ::: Http Async
     =>
     scope.race
       (-> fetch-from-primary)
       (-> fetch-from-replica)
       (-> fetch-from-cache)

   ;; supervised — failures isolated per child
   fn process-batch :: #[Job] #[Result] ::: Async Log
     => jobs
     scope.supervised s
       jobs |> @ (j -> s.fork (-> process j))
            |> @ await

   ;; escape hatch: fire-and-forget (use sparingly!)
   ;; the ! suffix signals this breaks structured concurrency
   fn notify-and-continue :: Event Response ::: Async
     => event
     detach! (-> send-analytics event)  ;; runs independently
     :ok                                ;; returns immediately
   #+end_example

** 4.3 Streams & Flows

   Inspired by Missionary's unification of discrete and continuous signals.

   #+begin_example
   ;; discrete stream (sequence of events)
   effect Stream a
     emit :: a -> ()
     done :: ()

   ;; continuous flow (always-current value, like a signal)
   effect Flow a
     sample :: () -> a

   ;; stream operations (lazy, pull-based by default)
   fn process-events :: #[Event] () ::: Async Log
     => events
     events
       |> /? (e -> e.type == :purchase)
       |> @ calculate-tax
       |> chunk 100
       |> @ batch-insert

   ;; flow combination (reactive)
   fn dashboard-state :: Flow User -> Flow #[Order] -> Flow Dashboard
     (user orders ->
       flow (-> make-dashboard (sample user) (sample orders)))
   #+end_example

** 4.4 Channels & Backpressure

   #+begin_example
   ;; bounded channel (backpressure built-in)
   ch := chan 64 :: Chan Event

   ;; send/receive are effects
   effect Chan a
     send :: a -> ()
     recv :: () -> a

   ;; select (multiplexing)
   select
     recv ch1 => (msg -> handle-primary msg)
     recv ch2 => (msg -> handle-fallback msg)
     timeout (Duration 5 :sec) => (-> log "timed out")
   #+end_example

* Part V: Choreographic Programming

** 5.1 Located Types

   Every value in a choreographic context carries a *location annotation*
   specifying which participant owns it. The compiler performs Endpoint
   Projection (EPP) to generate per-participant code automatically.

   #+begin_example
   ;; located type syntax: value @$ROLE
   ;; declares which participant holds the data

   fn buy-protocol :: (Str @$BUYER) (Result @$BUYER) ::: Choreo
     => title
     ;; $BUYER sends title to $SELLER
     title ~> $SELLER                         ;; communicate $BUYER->$SELLER

     ;; $SELLER looks up price, sends to $BUYER
     price := $SELLER.lookup title
     price ~> $BUYER

     ;; $BUYER decides
     decision := $BUYER.decide price
     match decision
       :buy =>
         decision ~> $SELLER                  ;; knowledge of choice
         $SELLER.ship title
         :ok @$BUYER
       :pass =>
         decision ~> $SELLER
         :cancelled @$BUYER
   #+end_example

** 5.2 Communication Primitives

   #+begin_example
   ;; ── Core choreographic operators ─────────────────────────
   ~>      ;; send: val @$A ~> $B            ($A sends to $B)
   <~      ;; receive (sugar for ~> other direction)
   ~*>     ;; broadcast: val @$A ~*> #[$B $C $D] ($A sends to all)
   ~/      ;; select: cond @$A ~/ $B         ($A informs $B of branch choice)

   ;; ── Located computation ──────────────────────────────────
   $ROLE.expr      ;; expr evaluated at $ROLE
   val @$ROLE      ;; located-at: in types (Str @$BUYER) and in
                   ;; expressions (match cmd @$PRIMARY dispatches on $PRIMARY)

   ;; ── Enclave (subset of participants) ─────────────────────
   enclave #[$A $B]
     ;; only $A and $B participate in this block
     shared := $A.compute-share
     shared ~> $B
     $B.verify shared
   #+end_example

** 5.3 Endpoint Projection (EPP)

   The compiler projects a single choreography into per-role programs.

   #+begin_example
   ;; The buy-protocol above compiles to:

   ;; ── $BUYER projection ────────────────────────────────────
   fn buy-protocol-BUYER :: () Result ::: Net IO
     =>
     title := input "What book?"
     send $SELLER title
     price := recv $SELLER
     decision := decide price
     match decision
       :buy =>
         send $SELLER :buy
         :ok
       :pass =>
         send $SELLER :pass
         :cancelled

   ;; ── $SELLER projection ───────────────────────────────────
   fn buy-protocol-SELLER :: () () ::: Net IO Db
     =>
     title := recv $BUYER
     price := lookup title
     send $BUYER price
     branch := recv $BUYER
     match branch
       :buy  => ship title
       :pass => ()
   #+end_example

** 5.4 Census Polymorphism & Dynamic Participants

   Based on Bates et al. (PLDI 2025): choreographies polymorphic over
   the number and identity of participants.

   *The problem:* Normal choreographies have a fixed, compile-time-known
   set of roles ($BUYER, $SELLER). But real distributed systems often need
   "do X for every worker" where the worker count is unknown at compile
   time — think map-reduce, gossip protocols, voting.

   *The solution:* Make the participant set a type parameter. =forall
   $WORKERS= means "this choreography works for any set of participants
   playing the $WORKER role." At runtime, =census $WORKERS= returns the
   current member set (a =std.choreo= function). The compiler projects
   this into: $COORD gets a loop of sends/receives; each $WORKER gets
   a "receive, compute, send back" program. The projection is
   *per-role-template*, instantiated at deployment time for each actual
   participant.

   *=par-each=* is the choreographic parallel iteration keyword. It
   iterates over a role set and runs the body once per participant,
   in parallel. Syntax: =results := par-each w <- $ROLE_SET body=.
   The last expression of each iteration is collected into a vector
   bound on the coordinating role. Inside the body, =w= refers to the
   current participant.

   #+begin_example
   ;; census-polymorphic: works for any set of $WORKERS
   fn map-reduce
     :: (a -> b) @$COORD
     -> #[a] @$COORD
     -> (b -> b -> b) @$COORD
     -> b @$COORD
     ::: Choreo
     forall $WORKERS
     => f data merge
     ;; $COORD splits data into as many chunks as there are $WORKERS
     chunks := $COORD.partition data (census $WORKERS)

     ;; each $WORKER receives their chunk, computes, sends back
     ;; par-each collects the last expression from each iteration
     results := par-each w <- $WORKERS
       chunk := chunks[w] ~> w       ;; $COORD sends chunk to worker w
       result := w.compute f chunk   ;; worker w computes locally
       result ~> $COORD              ;; worker w sends result to $COORD

     ;; $COORD combines all collected results
     $COORD.reduce merge results
   #+end_example

** 5.5 Fault Tolerance

   A =with= block may have an =on-failure= clause that runs when the
   primary body raises an unhandled exception or effect. The
   =on-failure= block is indented at the same level as =with= and
   executes in the same choreographic context (same roles, same
   communication topology).

   #+begin_example
   ;; choreography with failure handling
   fn safe-transfer :: () () ::: Choreo Async
     =>
     with choreo-retry {max= 3 backoff= :exponential}
       amount := $SENDER.input-amount
       amount ~> $RECEIVER
       ack := $RECEIVER.process amount
       ack ~> $SENDER

     on-failure
       $SENDER.rollback
       $RECEIVER.rollback
   #+end_example

** 5.6 Scope Semantics in Choreographies

   When =scope= appears inside a choreography, it obeys the same
   structured concurrency rules as elsewhere: the scope *blocks* until
   all forked fibers complete. This means code after the scope sees
   the completed results.

   #+begin_example
   ;; scope blocks: replication finishes BEFORE Ok is returned
   fn replicated-put :: Str Str (Reply @$CLIENT) ::: Choreo Async
     => k v
     $PRIMARY.store k v
     scope s
       s.fork (-> k ~> $REPLICA; v ~> $REPLICA; $REPLICA.store k v)
     ;; ↑ scope exits only after fork completes
     Ok @$CLIENT   ;; client gets Ok AFTER replication

   ;; if you want "reply first, replicate later" you need detach!
   fn fast-put :: Str Str (Reply @$CLIENT) ::: Choreo Async
     => k v
     $PRIMARY.store k v
     detach! (-> k ~> $REPLICA; v ~> $REPLICA; $REPLICA.store k v)
     Ok @$CLIENT   ;; client gets Ok IMMEDIATELY, replication continues
   #+end_example

* Part VI: Gradual Verification Spectrum

** 6.1 Level 0 — Dynamic (No Annotations)

   #+begin_example
   ;; completely dynamic, no types, no constraints
   fn process
     (data -> data |> transform |> validate |> save)
   #+end_example

** 6.2 Level 1 — Schemas (Runtime-Validated Specs)

   Irij uses a Malli/spec-inspired schema system instead of Hindley-Milner
   type inference. Schemas are data, serve multiple purposes (validation,
   generation, documentation, serialization), and compose like values.

   #+begin_example
   ;; schema declaration — one definition gives validation + generation + docs
   schema Person
     :name Str
     :age  (Int :min 0 :max 150)
     :role (Enum :admin :user :guest)

   ;; schema annotations on functions (optional but recommended for pub API)
   fn process :: Data -> Result
     (data -> data |> transform |> validate |> save)

   ;; partial annotation — mix annotated and unannotated
   fn transform :: Data -> _    ;; return shape unspecified
     (data -> ...)

   ;; schemas power Arbitrary generation for law checking
   fn sort
     law idempotent = forall (xs : #[Int]). sort (sort xs) == sort xs
     ...
   #+end_example

   Note: Level 0 and Level 1 have the same function *body* (a lambda).
   The only difference is the optional =:: Data -> Result= annotation.
   Schemas are checked at runtime, not compile time. No HM inference —
   the language relies on contracts, laws, and effects for verification.

** 6.3 Level 2 — Contracts (Runtime Checks)

   Contracts add runtime-checked preconditions and postconditions to
   functions. Two forms exist, serving different scopes:

   *=pre=/=post=: function-level contracts.* Checked on every call.
   =pre= predicates receive the function's arguments; =post= receives
   the return value. A violation is always the caller's fault (for =pre=)
   or the implementation's fault (for =post=).

   *=contract=/=in=/=out=: module-boundary contracts with blame tracking.*
   Used on =pub= functions — the API surface. When a contract is violated,
   *blame tracking* identifies which party broke the agreement:

   - =in= violation → the *caller* (outside module) sent bad data.
     Error message: "module X violated api-endpoint's input contract"
   - =out= violation → the *implementation* (this module) returned bad data.
     Error message: "api-endpoint violated its output contract to module X"

   This distinction matters in large codebases (or multi-LLM systems)
   where the caller and the implementer are different parties. It comes
   from Racket's contract system (Findler & Felleisen, 2002).

   #+begin_example
   ;; ── Function-level contracts ─────────────────────────────
   fn withdraw :: Account -> Amount -> Account
     pre  (a -> a.balance >= 0)
     pre  (a amt -> amt > 0 && amt <= a.balance)
     post (result -> result.balance >= 0)
     (account amount ->
       {...account balance= account.balance - amount})

   ;; ── Module-boundary contracts (with blame tracking) ──────
   pub fn api-endpoint :: Request -> Response
     contract
       in  (r -> valid-json? r.body)      ;; caller's fault if violated
       out (r -> r.status < 500)          ;; our fault if violated
   #+end_example

** 6.4 Level 3 — Property Testing (Algebraic Laws)

   This is the *key innovation*: first-class law declarations that are
   automatically tested via property-based testing and optionally
   verified via SMT.

   #+begin_example
   ;; declaring a typeclass/protocol with laws
   proto Monoid a
     empty :: a
     append :: a -> a -> a

     law identity-left  = forall x. append empty x == x
     law identity-right = forall x. append x empty == x
     law associative    = forall x y z.
       append (append x y) z == append x (append y z)

   ;; implementing with automatic law checking
   impl Monoid for Str
     empty := ""
     append := (++)
     ;; compiler runs QuickCheck-style tests on all three laws
     ;; at compile time or in CI

   ;; custom laws on any function
   fn sort :: Ord a => #[a] -> #[a]
     law idempotent   = forall xs. sort (sort xs) == sort xs
     law permutation  = forall xs. (sort xs |> /# ) == (xs |> /#)
     law ordered      = forall xs. sorted? (sort xs)
     ...
   #+end_example

** 6.5 Level 4 — Refinement Types

   #+begin_example
   ;; refinement type (SMT-verified at compile time)
   type Nat := {x :: Int | x >= 0}
   type NonEmpty a := {xs :: #[a] | /# xs > 0}
   type Percentage := {x :: Float | x >= 0.0 && x <= 100.0}

   ;; function with refinement
   fn head :: NonEmpty a -> a
     (#[x ..._ ] -> x)

   ;; the compiler statically proves this is safe:
   fn safe-example :: #[Int] -> Maybe Int
     => xs
     if /# xs > 0
       Just (head xs)    ;; OK: compiler sees /# xs > 0, refines to NonEmpty
     else
       Nothing
   #+end_example

** 6.6 Level 5 — Proofs (Dependent Types)

   #+begin_example
   ;; full dependent types for critical sections
   fn matrix-mul
     :: Matrix m n -> Matrix n p -> Matrix m p
     proof dimensions-match
     ...

   ;; theorem as a type
   type AppendAssoc a (xs ys zs :: #[a])
     := append (append xs ys) zs == append xs (append ys zs)
   #+end_example

* Part VII: Content-Addressed Code

** 7.1 Hash-Based Identity

   #+begin_example
   ;; every definition has a hash identity (SHA3-512)
   ;; names are just metadata pointers to hashes

   fn factorial :: Int -> Int  ;; hash: #a7f3b2c1...
     0 => 1
     n => n * factorial (n - 1)

   ;; renaming is free (just update name -> hash mapping)
   ;; no builds needed (unchanged hashes use cached results)
   ;; no dependency conflicts (deps are hashes, not versions)
   #+end_example

** 7.2 Effect-Aware Hashing (Novel)

   Irij extends content-addressing to include effect signatures:

   #+begin_example
   ;; hash captures: AST structure + type signature + effect row
   ;; two functions with same body but different effects get different hashes

   fn read-config :: () Config ::: FileIO       ;; hash: #x1...
   fn read-config-test :: () -> Config           ;; hash: #x2... (different!)

   ;; enables: effect-polymorphic caching
   ;; if a pure function was called with same args before, skip recomputation
   ;; effect-annotated functions are only cached when handler permits it
   #+end_example

* Part XI: Metaprogramming — Deferred by Design

  *Decision: defer macros/metaprogramming to Phase 3+.*

  Rationale: Macros are the #1 source of confusing error messages in
  Lisp-family languages and would dramatically complicate LLM code
  generation. For a token-efficient language, the traditional uses of macros
  are better served by other mechanisms:

  - *Boilerplate elimination:* the LLM generates expanded code directly.
  - *DSLs:* algebraic effects + handlers provide composable abstraction.
  - *Code transformation:* the content-addressed codebase stores ASTs
    that tooling can transform externally.

  When metaprogramming is eventually added, the approach will be:

  #+begin_example
  ;; A "macro" is a function from AST-hash to AST-hash.
  ;; Since all code is content-addressed, transformation is just
  ;; a function that reads one hash and writes another.

  ;; This will be explored in Phase 3, informed by real usage patterns.
  ;; Until then, prefer effects + handlers + LLM generation.
  #+end_example

* Part XIII: Open Questions & Risks

** 13.1 Technical Risks

   - *Effect compilation on JVM.* Algebraic effects need delimited
     continuations. Project Loom provides virtual threads but not full
     one-shot continuations. Options: (a) CPS transform at compile time,
     (b) use Loom's =Continuation= internal API, (c) implement on Truffle
     which has native continuation support. *Recommendation: start with
     CPS transform, migrate to Loom continuations as they stabilize.*

   - *Choreographic EPP complexity.* Projection of higher-order
     choreographies with effects is an open research problem. Klor and
     HasChor handle the simple cases; complex interactions may need
     manual annotations. *Recommendation: start with first-order
     choreographies, extend incrementally.*

   - *Gradual verification overhead.* Typed Racket showed up to 235×
     overhead at type boundaries. *Mitigation: JIT compilation
     eliminates 90%+ overhead (Pycket result), use shallow types at
     boundaries by default.*

   - *Content-addressed code + mutation.* Mutable bindings complicate
     hashing. *Mitigation: hash the pure computation; effects (including
     mutation) are tracked separately in the effect row.*

** 13.2 Adoption Risks

   - *Zero training data for LLMs.* Novel syntax means LLMs have no
     pre-training. See Part XIV for the multi-representation training
     strategy that mitigates this.

   - *Learning curve.* Effects + choreography + gradual types = many new
     concepts. *Mitigation: Level 0 (fully dynamic, no annotations) is
     the default entry point. Features are opt-in, incremental.*

   - *JVM startup time.* Clojure's notorious slow startup.
     *Mitigation: GraalVM Native Image (~50ms startup) or CRaC
     (checkpoint/restore) from day one.*

** 13.3 Design Decisions Still Open

   - *Eager vs Lazy evaluation.* Lazy is more token-efficient (no
     explicit delay/force) but harder to reason about effects.
     *Leaning: strict by default, lazy via explicit =lazy= / streams.*

   - *Row polymorphism implementation.* Koka-style evidence passing
     vs. dictionary passing vs. Effekt-style capability passing.
     *Needs prototyping.*

   - *Syntax for type application.* Explicit =[Int]= vs. inferred
     everywhere vs. turbofish =::<Int>=.
     *Leaning: inferred, with explicit =@Int= when needed.*

   - *Error model.* Checked effects (Err in effect row) vs. unchecked
     panics vs. both. *Leaning: both, like Rust's Result + panic.*

   - *Record system.* Nominal (declared types) vs. structural
     (anonymous) vs. both. *Leaning: both, with structural subtyping
     for anonymous records.*

* Part XIV: LLM Training Strategy — Multi-Representation Learning

  *Core question:* If indentation is the canonical form, how do LLMs
  learn to generate Irij with zero pre-existing training data?

  *Answer:* Train on multiple representations, generate in one.

** 14.1 The Transfer Learning Insight

   LLMs don't learn languages from scratch — they transfer structural
   knowledge from languages they already know. The key findings:

   - MIT experiment (2024): showed an LLM just a few examples of a novel
     syntax and it could reliably generate code in that syntax. Few-shot
     transfer works for syntactic patterns.

   - BPE tokenization means the *tokens* of Irij are already in the LLM's
     vocabulary: =fn=, =:==, =|>=, =#[=, =match=, etc. are all standard
     ASCII sequences. The LLM doesn't see "a new language" — it sees
     familiar token sequences in a new grammatical arrangement.

   - Indented block structure transfers from Python, Haskell, YAML, and
     every markdown code block the LLM has ever seen. Pipeline operators
     transfer from Elixir, F#, OCaml. Pattern matching transfers from
     Haskell, Rust, Scala. Effect syntax is novel, but structurally
     resembles Java's =throws= clauses.

** 14.2 Multi-Representation Training Corpus

   The training strategy uses three representations of the same code:

   #+begin_example
   ;; ── Representation 1: Natural language specification ─────
   ;; "A function that fetches a user by ID from the database,
   ;;  returns Ok with the user or Err if not found.
   ;;  Requires database read access and logging."

   ;; ── Representation 2: Irij canonical (indented) form ────────
   fn fetch-user :: Id (Result User NotFound) ::: DbRead Log
     => id
     log "fetching user ${id}"
     result := db.query "SELECT * FROM users WHERE id = ?" id
     match result
       #[row ..._ ] => Ok (row->user row)
       #[]          => Err NotFound

   ;; ── Representation 3: Irij S-expression (interchange) form ──
   (fn fetch-user (:: Id (Result User NotFound) ::: DbRead Log)
     (=> id (do (log "fetching user ${id}")
       (:= result (db.query "SELECT * FROM users WHERE id = ?" id))
       (match result (=> (#[row ..._]) (Ok (row->user row)))
                     (=> (#[]) (Err NotFound))))))
   #+end_example

   Training on all three forms teaches the LLM:
   - Spec → Code translation (from natural language)
   - Indented ↔ S-expr equivalence (structural understanding)
   - Indented form as the preferred generation target

** 14.3 The Bootstrapping Pipeline

   1. *Phase A (synthetic):* Use existing LLMs to translate a large
      corpus of Clojure/Haskell/Koka code into Irij syntax, manually
      correcting errors. Yields ~10K validated examples.

   2. *Phase B (compiler-validated):* Feed examples to LLMs, have them
      generate new Irij code, run the Irij compiler for type/effect checking,
      keep only code that compiles. Yields ~100K validated examples.

   3. *Phase C (fine-tune):* Fine-tune a model on the validated corpus.
      The fine-tuned model generates directly in Irij indented form.

   4. *Phase D (RLHF/DPO):* Use compiler feedback as reward signal.
      Code that type-checks, effect-checks, and passes law verification
      gets positive reward. This creates a model that inherently respects
      the type system.

   The S-expression form is valuable during Phase A (Lisp knowledge
   transfers to S-expr generation), but by Phase C the model generates
   indented form directly because that's what the validated corpus
   predominantly contains.

* Appendix B: Full Example — Distributed Key-Value Store

  #+begin_example
  mod demo.kv-store

  use std.choreo
  use std.async
  use std.collection {HashMap}
  use std.net
  use std.core {loop}

  ;; ── Roles ─────────────────────────────────────────────
  role $CLIENT
  role $PRIMARY
  role $REPLICA

  ;; ── Data ──────────────────────────────────────────────
  type Command
    Get Str
    Put Str Str
    Del Str

  type Reply
    Found Str
    NotFound
    Ok

  ;; ── Choreography ──────────────────────────────────────
  fn kv-protocol :: (Command @$CLIENT) (Reply @$CLIENT) ::: Choreo Async
    => cmd
    cmd ~> $PRIMARY

    match cmd @$PRIMARY
      Put k v =>
        cmd ~/ $REPLICA          ;; inform replica of branch
        $PRIMARY.store k v
        ;; replicate: scope blocks until fork completes
        scope s
          s.fork (->
            cmd ~> $REPLICA
            $REPLICA.store k v)
        Ok @$CLIENT

      Get k =>
        cmd ~/ $REPLICA
        result := $PRIMARY.lookup k
        result ~> $CLIENT
        match result @$CLIENT
          Just v  => Found v
          Nothing => NotFound

      Del k =>
        cmd ~/ $REPLICA
        $PRIMARY.delete k
        scope s
          s.fork (-> cmd ~> $REPLICA; $REPLICA.delete k)
        Ok @$CLIENT

  ;; ── Entry Point ───────────────────────────────────────
  fn main :: () () ::: IO
    =>
    config := read-config ()
    with (tcp-transport config)
      with (console-to-stdout)
        ;; loop is a std.core function: calls thunk forever
        loop (->
          cmd := $CLIENT.prompt
          reply := kv-protocol cmd
          $CLIENT.display reply)
  #+end_example

* Appendix C: Interactive Development & REPL Roadmap

  ℑ takes inspiration from Clojure's REPL-driven development model.
  The vision is a connected REPL that allows redefining functions in a
  running application, enabling truly interactive development.

  *Current (Phase 2):* Basic REPL with evaluate-and-print, file loading,
  history, multi-line input. Top-level bindings can be redefined in a
  session.

  *Planned capabilities:*
  - *Hot function redefinition:* Redefine =fn= declarations in a running
    program. Closures that captured the old definition continue with old
    behavior; new calls use the new definition.
  - *Connected REPL:* Attach to a running Irij process over a socket/pipe,
    evaluate code in its environment (like nREPL).
  - *Namespace-aware reload:* =:reload mod.name= re-evaluates a module,
    replacing its bindings in the running environment.
  - *State inspection:* =:env= to show current bindings, =:type expr= to
    query inferred types.

  *Design constraint:* The =Environment= uses =define()= which silently
  overwrites existing bindings. This is intentional — the REPL must allow
  redefinition. Phase 4 (modules) will add =pub= visibility but the REPL
  will always bypass module boundaries for development access.
