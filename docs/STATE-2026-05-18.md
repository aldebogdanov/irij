# Irij — State of the Project, 2026-05-18

A blunt, end-of-v0.5.0 audit. No marketing. What works, what doesn't,
what's stale, what to do next.

## 1. Headline numbers

- **Version**: v0.5.0 (just shipped).
- **Java SLOC**: 18,136. Interpreter 6,274 (35%), compiler 6,051 (33%),
  rest is parser/CLI/nREPL/MCP/AST.
- **Heaviest files**: `Interpreter.java` (3,632 LOC),
  `ClassEmitter.java` (3,588 LOC). These two are 40% of all Java.
- **Tests**: 865 Java `@Test` methods across 24 classes, all green.
  299 integration tests across 19 `tests/test-*.irj` files, all green.
- **Stdlib (Irij)**: 16 modules — `collection convert datastar db env
  fs func http jvm list math random serve test text time`.
- **Examples**: 12 `.irj` files in `examples/`.
- **Active branches**: 1 (`main`). 7 leftover topic branches (see §4).

## 2. What is implemented (honest reading)

### Solid
- Parser + lexer (ANTLR4, Python-style INDENT/DEDENT).
- Interpreter — fully covers the language as far as the spec is
  *implemented*. Reference implementation.
- Bytecode compiler — covers everything the interpreter does at the
  user-visible language level: literals, fns, lambdas, closures,
  match, all collection literals, protocols/impls, modules
  (compile-time inlining), Java interop, hot redef via invokedynamic,
  structured concurrency on virtual threads, algebraic effects in two
  lowerings (threaded + state-machine).
- Effect system — declared rows, runtime stack, two lowerings,
  compile-time `EffectRowChecker` lint, subsumption at call sites,
  per-ref JVM capability propagation, polymorphism via `::: Any`.
- Specs — primitive / composite (App, VecSpec, SetSpec, TupleSpec) /
  arrow / enum / user-declared product+sum. Validate both inputs and
  outputs at fn boundaries in both interp and bytecode. clinit-time
  registration in compiled classes.
- Contracts — `pre`, `post`, `in`, `out` validated at fn entry/tail
  in both interp and bytecode (v0.5.0). Blame text matches.
- Module system — `mod`/`use :open`, qualified access, compile-time
  inlining for bytecode, in-process registry for interp.
- Package management — `irij install` from registry, `irij publish`,
  git deps via `irij.toml`/`deps.irj`.
- Tooling — REPL (`irij repl`), nREPL server (`eval` interp, +
  `eval-bytecode` with cross-eval namespace), Emacs client,
  Datastar-based playground at irij.online, MCP server.
- Java interop — `Class/member`, `Class/new`, dot-access, capability
  via `::: JVM` (plus per-ref refinement).
- Build/deploy — `irij build` produces a self-contained runnable jar
  in three modes (`interp`, `bytecode-threaded`, `bytecode-sm`);
  optional embedded nREPL via `--nrepl-port=N`.

### Stubbed / parsed-but-not-implemented
- **Choreography** — parser accepts `@$ROLE`, `~>`, `<~`, `~*>`,
  `~/`, `par-each`, `forall`, located types. Interpreter throws
  "Choreographic programming not yet implemented" on any
  `Expr.ChoreoExpr`. Compiler has no path at all. The spec.org puts
  choreography front and centre as a defining feature — that's
  aspirational, not delivered.
- **Capabilities (`cap`)** — reserved word, parser hook, runtime
  no-op.
- **Refinement / dependent types** — not started.
- **Channels (`send`/`recv`/`select`)** — not started; concurrency
  is fibers + `par`/`race`/`scope` only.
- **Streams / Flows** — not started.
- **LSP server** — not started.
- **GraalVM native image** — not configured.
- **Content-addressed code** — not started.

### Interpreter-only (i.e. real gap between modes)
After v0.5.0 the bytecode mode catches up on most fronts. What
still lives only in the interpreter:

1. **Law verification** (`law name :: T  forall x. P x` /
   QuickCheck-style). Needs Arbitrary instances and a sample loop;
   neither is wired into bytecode mode.
2. **Module-boundary blame attribution** for `in`/`out` contracts.
   Bytecode emits the generic "Input contract violated in 'f'"
   message without the caller-side / module-prefix variant. Cosmetic
   but visible.
3. **Sandboxed nREPL `eval`** — the playground session loop calls
   `Interpreter.evalString` directly. `eval-bytecode` is parallel
   but stateless for fns (top-level `:=` binds persist, fn defs
   don't cross evals — see `docs/internals/nrepl.md`).
4. **Builtins coverage** — 91 builtins are registered in
   `Builtins.java`; the bytecode emitter has fast paths for ~47.
   The rest are still reachable from compiled code via stdlib
   inlining + `RuntimeSupport.callAny`, but anything that bypasses
   stdlib and calls a builtin directly won't have the fast path.

## 5. Interpreter removal — feasibility

The user asked: "get rid of interpreter completely." Honest answer:
not in one session, not in one release. Here's the breakdown.

### What blocks today

1. **CLI defaults route to interpreter.** `irij run`, `irij eval`,
   `irij test`, `irij repl`, default `irij build` mode — all interp.
   Switching defaults is mechanical; making bytecode the *only*
   path is not.
2. **nREPL primary `eval` op uses the interpreter.** The playground,
   the Emacs client and the embedded-in-jar nREPL all depend on it.
   `eval-bytecode` exists but doesn't yet support cross-eval fn
   defs.
3. **Laws + Arbitrary** — pure-interp feature today.
4. **Module-boundary blame** — pure-interp.
5. **91 builtins registered in `Interpreter`/`Builtins.java`**, some
   are interpreter-only (DB tx callback, nREPL eval re-entrance,
   raw-multipart-save against in-process state, etc.).
6. **Top-level scripts** — interpreter executes statements in
   sequence with a live `globalEnv`. Bytecode emits a `main()`. Some
   stdlib modules + tests assume top-level eval semantics that the
   bytecode path doesn't currently match 1-for-1 (mostly around
   forward-reference order and dynamic-fn-redef behaviour).

### Phased path (estimate: 3 months)

- **Phase R1 — Bytecode parity for laws.** Implement `Arbitrary`
  registry + `run-law` driver in `RuntimeSupport`. Emit one method
  per law in the compiled class, call from a `__irij_run_laws`
  hook. ~1 week.
- **Phase R2 — Cross-eval fn defs in nREPL bytecode mode.** Per the
  design already sketched in `docs/internals/nrepl.md` §"What's
  still not cross-eval". Emit each fn with a clinit hook that
  registers an `IrijFn` wrapper in the session's namespace map.
  `emitVarLoad` already falls back through `nsGet`. ~1–2 weeks.

  Subtask done (v0.5.0+): bytecode-mode `ModuleInliner` now accepts
  resolved seed directories from `DependencyResolver`. `irij build
  --mode=bytecode-sm` can inline `use vrata.html` and other seed
  modules; previously it could only see classpath + project root.
- **Phase R3 — Migrate the remaining 44 builtins.** Some are
  trivial (one Java method); some (DB transactions with callbacks,
  nREPL re-entrance, multipart handling) need careful design. ~3
  weeks. The blocker right now is that `Builtins.install(Environment)`
  is coupled to the interpreter's `Environment` type; bytecode mode
  has no entry point to populate them. Likely refactor: split each
  builtin into a static method on a runtime helper class (mirroring
  `RuntimeSupport`), and emit a per-name fast path on the
  emitter side. Concrete first dominoes: `replace`, `substring`,
  `split`, `join`, `assoc`, `get`, `dissoc`, `index-of`, `keys`,
  `vals`, `contains?` — these are the heavy hitters appearing
  across `std/*.irj` and real apps.
- **Phase R4 — Make bytecode the default for `run`/`test`/`eval`.**
  Keep `--mode=interp` as fallback. Ship as v0.6.0. ~1 week to
  flip + chase regressions.
- **Phase R5 — Remove the interpreter.** Delete `dev.irij.interpreter`
  package. Cut ~6,000 LOC. Replace REPL with an indy-cached
  bytecode REPL. ~1–2 weeks to delete + clean up.

Risks:
- The interpreter is the *reference* implementation. Most semantic
  questions ("what does X do?") are answered by reading
  `Interpreter.java`. Removing it makes the bytecode emitter the
  reference, which has a much higher cognitive cost (3,600 LOC of
  ASM-emitting Java).
- Module-boundary blame, hot-redef interaction with bytecode, and
  effect-row TCO bypass have observable differences. Each needs
  test coverage before deletion.

Recommendation: do R1–R3 (parity), ship v0.6.0 with bytecode as
default, then make the interpreter-removal call after one release
cycle of users running on bytecode-only by choice.

## 6. Forward roadmap — libs + apps to build with Irij

Two horizons. Pick from each.

### Libraries / seeds to publish (close gaps users hit)

1. **`std.auth`** — JWT, password hashing (Argon2), session tokens,
   OAuth2 client. The website already needs this; everyone else
   will too.
2. **`std.queue`** — abstraction over Redis Streams + SQLite
   (default-queue handler), with `enqueue`/`dequeue`/`ack` ops.
3. **`std.s3`** — generic object-store effect with handlers for AWS
   S3, R2, MinIO. JVM SDK underneath.
4. **`std.email`** — SMTP send + IMAP read effect, default-smtp
   handler.
5. **`std.cache`** — in-process LRU + Redis handler, effect-based.
6. **`std.crypto`** — AEAD, hashing, KDF, symmetric/asymmetric.
   Backed by JDK + Bouncy Castle.
7. **`std.test.snapshot`** — golden-file diffing for serialised
   values.
8. **`std.scheduler`** — cron-style scheduling effect built on
   `Timer` + structured concurrency.
9. **`std.csv`, `std.toml`, `std.yaml`** — fill out config/data
   parsers (we have JSON only).
10. **`std.observability`** — OpenTelemetry tracing/metrics effect
    + handler. Spans cross fiber boundaries.
11. **`std.template`** — server-side templating (Vrata is one
    answer; Mustache-style is another).

### Apps that could attract users

1. **Irij Package Registry → multi-tenant.** Today `irij.online`
   hosts one. Add namespacing, tokens, web auth. This is the
   shortest path to "people are using Irij as a real package
   system" — Cargo/npm/Hex moment.
2. **`vrata-admin`** — admin dashboard generator. Define a
   `spec` + a handler stack, get CRUD + auth + audit log out of
   the box. Plays directly to Irij's effect-row + spec strengths.
3. **`irij-bench`** — public benchmark suite + dashboard at
   `irij.online/bench`. Compare bytecode-sm vs threaded vs
   Clojure vs Python for fib/tak/JSON/HTTP throughput. Marketing
   muscle.
4. **`irij-lsp`** — Language Server. Hover, go-to-def,
   spec-driven autocomplete, effect-row diagnostics. Without this
   Irij stays an Emacs/CLI tool. Cursor/VS Code support is table
   stakes for adoption.
5. **`vrata-chat`** — minimal Slack-like chat over WebSockets +
   Datastar. Show off effects + structured concurrency on a
   real-time app.
6. **`irij-bot`** — Telegram/Discord bot framework with handler-
   based command routing. Easy showcase + viral.
7. **Reposado** — already deployed (the irij.online host).
   Document it as a reusable "Nix-deployed Irij + Caddy" template
   so others can ship Irij sites.
8. **Tutorial book** — `Learn Irij In Y Minutes`, then a longer
   handbook. People adopt languages they can read about.

### One-shot demos worth doing

- An effect-system tutorial that ports a 200-line Clojure async
  example into 80 lines of Irij with explicit effects + spec.
- A "build a Discord clone in 500 lines" stream/blog.
- The `bench/` dir already has fib+tak; add JSON encode/decode,
  HTTP echo, prime sieve, ackermann. Publish a chart.
